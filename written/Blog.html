Using the Daily.co API to implement a "Raise Your Hand" Feature

Suggest Edits

TLDR: This application's source code can be found on GitHub. Take it for a spin and dig around yourself.

What is Daily.co

Daily.co is a platform that makes it easy for developers to integrate video chat into any application. The Daily.co API allows you to integrate beautiful and intuitive video chats in your application with just a few lines of code. You can check it out with a free Daily.co account and API key to create your own meeting links to share with whoever you like.

The “Raise Your Hand” Feature

A "raise your hand" feature can be a useful piece of UX in group calls under many circumstances. Maybe you have a community call where a moderator needs to interact with call participants or maybe you have an online classroom where a student needs to get their teacher's attention.

While Daily.co gets you up and running with many built-in features, such as chat and screen-sharing, there are additional features one might want in an application that require a little bit of implementation. Luckily Daily.co's API gives you everything you need to customize your application's video experience.

What We Are Building

Here is an overview of what we want our feature to look like:

  • Each participant in the call will have their own virtual hand and a respective "raise your hand" button allowing them to change their hand state ("raised" or "lowered")- visible only to them. By design, this button will be outside the video call frame.
  • Each user view will contain an updating list of the participants currently in the call and whether they have their virtual hand raised or not. By design, this list will be outside the video call frame.
  • Everyone's hand state will be visible to everyone else in the call.

Note: Since we are expecting a very small number of participants and this app is mainly for demo purposes we will not turn off chat, microphone, or camera functionality by default. Read more about why you might want to do this below.

Building the Application

The main entry point for the library's functionality is the DailyIframe class. We initialize our video call iframe using the .createFrame(parentEl, properties) factory method and then join our room via the .join(properties) instance method.

  • JavaScript
    let room = { url: "https://reina.daily.co/raise-hand-demo" };

app.js

First, you have to create a Daily.co room. There are many ways to do this:

  1. Via https://dashboard.daily.co/
  2. Via a (Daily.co helper script)[https://github.com/daily-co/daily-demos/blob/master/static-demos/shared-assets/create-demo-room.js] to create a demo room under the api-demo.daily.co domain.
  3. Via the API docs
  1. Via your own script, for example:
  • JavaScript
var request = require("request");

var options = {
  method: 'POST',
  url: 'https://api.daily.co/v1/rooms/room_name',
  headers: {'content-type': 'application/json', authorization: 'Bearer '}
};

request(options, function (error, response, body) {
  if (error) throw new Error(error);

  console.log(body);
});

Next, we can define our destroy() function pretty easily. This will be called when the user presses the "leave" button. This will toggle a couple of inactive divs out of view and toggle into view a simple goodbye message.

  • JavaScript
function destroy(){
    document.querySelector(".main-section").classList.toggle("hide");
    document.querySelector(".start-button").classList.toggle("hide");
    document.querySelector(".destroy-view").classList.toggle("hide");
    callFrame.destroy();

}

We will also want to keep handy our showEvent helper function. While not necessary, I recommend you poke around and see what this function prints out as events are fired off. In the image below, we see what is returned when participant-joined fires off. Soon we will set up our listener for this exact event (among others) to keep track of the participants in our room.

  • JavaScript
function showEvent(e) { console.log('callFrame event', e) }

Let's write our main driving function. Earlier we touched upon the many ways we could create a room and here is where we need to use that URL. I passed in an appropriately named room to get us started and initialized our UI by bringing some elements into view. Also, take note of current - this variable will help us keep track of the hand state. Here we also want to set up our event listeners and hook up our callback functions.

  • JavaScript
let current = {
    id: '',
    hand: false
}

async function start() {

    let room = { url: "https://reina.daily.co/raise-hand-demo" };

    // create a video call iframe and add it to document.body
    // defaults to floating window in the lower right-hand corner
    // alternatively you could create iframe in index.html and then wrap it
    //
    window.callFrame = window.DailyIframe.createFrame(); //constructor
    window.callFrame.join({ url: room.url, showLeaveButton: true }); //top level props



    // initialize user view: hand status button and prepare participant list
    //
    document.querySelector(".hand-section").classList.toggle("hide");
    document.querySelector(".guest-section").classList.toggle("hide");
    document.querySelector(".start button").classList.toggle("hide"); // uncomment before pushing

  	// set up listeners and add callbacks
    //showEvent useful for debugging/learning purposes
    //
    callFrame.on('loaded', showEvent)
    .on('started-camera', showEvent)
    .on('camera-error', showEvent)
    .on('joining-meeting', showEvent)
    .on('error', showEvent)
    .on('joined-meeting', updateParticipantView)
    .on('participant-joined', updateParticipantView)
    .on('participant-updated', showEvent)
    .on('participant-left', deleteParticipant)
    .on('left-meeting', destroy)
    .on('app-message', msg);
}

There are two events we want to listen for to know when to update our participant list: joined-meeting and participant-joined. We will be using the data returned by these events to keep track of participants (local and others) and state. We are listening for these two events via a nice switch statement.

When our "local" user joins the call we want to append him to our view via our addParticipant() function. When another participant joins we want to inform all views connected to our room via a message to update the list.

  • JavaScript
async function updateParticipantView (e) {
    console.log(showEvent(e))
    switch (e.action) {
        case 'joined-meeting' :
            let temp = callFrame.participants().local.user_id;
            addParticipant(temp, 'local');
            current.id = temp;
        case 'participant-joined' :
            addParticipant(e.participant.user_id, 'participant');
            callFrame.sendAppMessage(current);
        }
}

Below we have our function that makes an info block for each participant- appending to our meeting-participants-info div. It is important to note that as we append our participant blocks we are also generating an id using their user_id property. HTML standards say that we cannot have id's begin with numbers so that is why we are prefixing the ID's with 'who-' and 'hand-'. We added some inline styling to make the view easier on the eyes.

  • JavaScript
async function addParticipant(id, role) {
    let infoList = document.getElementById('meeting-participants-info');

    // upon join default is hand down
    // added border styling- easier to look at
    let infoEl =
    `<div style="border-style: dotted; padding: 10px; margin:10px;" id="who-${id}">
        <p class="type">${role}</p>
        <p class="hand" id="hand-${id}">false</p>
    </div>`;
    infoList.innerHTML += infoEl;
}

The most difficult part about this feature is that you have to create a way to share participant states between all possible room connections. When someone raises or lowers their hand, everyone should know. The way we do this is by communicating via 'app-message'. Without this event, it would be much more difficult to share variable states between session participants.

  • JavaScript
async function msg (e){
    changeHandState(e.data.id, e.data.hand);
}

Next, we have our changeHandState function that is connected to our messaging function as well as a button on index.html. Here is the logic updating hand movements (whether a participant raises their hand or not).

  • JavaScript
async function changeHandState(id, hand){

    let tempid, temphand;

    console.log(current);
    console.log(id == undefined);
    console.log(hand == undefined);

    if (id == undefined){
        tempid = current.id;
        temphand = current.hand;
        callFrame.sendAppMessage(current);
        current.hand = !current.hand;
    }
    else {
        tempid = id;
        temphand = hand;
    }


    let update = document.querySelector("#meeting-participants-info #hand-" + tempid);
    console.log(update);

    if (hand){
        update.innerHTML = temphand;
    }
    else {
        update.innerHTML = temphand;
    }
}

Finally, we want to create a simple function that helps us remove a participant from the view once they they are no longer active in the session. This can be achieved simply by calling .remove() on the HTML element containing the id of the element we want to remove.

  • JavaScript
async function deleteParticipant (e) {
    let this_id = e.participant.user_id;
    document.getElementById('who-' + this_id).remove();
}

Scalability

Technical considerations for scalable design and implementation

This application is meant to be used as a stepping stone but depending on your technical needs you have some design considerations to keep in mind. Building an application for a few users is different than for tens as is different for hundreds.

From a technical standpoint, the Daily.co API supports up to 10,000 programmatic rooms and each room up to 200 participants. Daily.co video is built on top of a standard called WebRTC. WebRTC allows you to send streams of data (such as video and audio) between any two computers on the Internet. In this case, we are using WebRTC inside the browser for video capabilities- this means that in a large room where everyone has turned on camera and microphone functionality, the browser will undoubtedly be overloaded.

Unfortunately, in this case, "a large room" is not in fact that large- Daily.co advises that only 6-8 cameras be on at any given time to give your application a standing chance. This number varies depending on variables like devices and networks but it is a good number to keep in mind nonetheless.

The following are useful considerations:

  • Turn on broadcast mode via the owner_only_broadcast property and set the is_owner property where applicable. Daily.co's owner_only_broadcast property enables only meeting owners to send audio and video. Turning on this setting would disable camera, microphone, and screen sharing functionality for others and will not display their respective buttons in the menu bar.
  • Make it so that when a user joins a room by default their video and audio are off- programmatically turning on their camera only in the event that they "raise their hand" and off when their turn has finished.
  • Turn off chat functionality but allow camera functionality.
